競程好難 - JOISC 2022 Day4 Reconstruction Project

Algorithm
Word count: 717Reading time: 2 min
 2023/05/24 

這題是我在選訓戳的，那時候跟 PixelCat 一起 vir，然後被揍得體無完膚。
過了超久之後，重新開始學習競程（？

題目

Judge 跟 PDF <- 揍我

給 \((N <= 500)\) 個點，\((M <= 100000)\) 條邊，每條邊有一個寬度。
共有 \((Q <= 1000000)\) 筆詢問，每次給一個指定寬度，
求最小生成樹的邊權和，每條邊的邊權是原寬度跟指定寬度的距離。

思路

一開始的想法

假設指定寬度是 \(0\)，就直接找一般的最小生成樹就好。
這時試著增加指定寬度，發現最小生成樹長得不一樣了，那是怎麼改變的？

一般找最小生成樹有一種方法是，將邊全部 sort 後，由小到大加入，
如果邊的兩端點已經聯通，就跳過。

運用同樣的想法，就可以找出每條邊的生命週期了，
一條邊要加入的時機是，邊權比他小的邊集合沒辦法讓這條邊的兩端點聯通，
反之亦然。

之後再觀察一下指定寬度與邊權大小的關係，就可以找出每條邊加入和拔掉的時機了。

我覺得蠻重要的觀察

但醬每條邊要做兩次，超麻煩，有沒有更好的做法？
稍微想一下就會發現，在樹上加入一條邊，同時也得拔掉另一條邊。
那這條邊是誰？

將邊照寬度由小到大 sort 後，假設第 \(i\) 條邊連接 \(A\)、\(B\) 點，
從第 \(i-1\) 條邊開始往回走，把路上的邊都加入邊集合，當 \(A\)、\(B\) 聯通時就停下來。
最後的那條邊就是要拔掉的那條。

為什麼是對的？

我們現在知道的是，當第 \(i\) 條邊的邊權小於最後那條邊時，第 \(i\) 條邊就要加入，
但最後那條邊會不會更早就死掉了？

答案是不會。

假設最後那條邊是第 \(k\) 條，代表從 \(k\) 到 \(i-1\) 剛好可以把 \(A\)、\(B\) 兩點連起來。
如果 \(k\) 這條邊在碰到 \(i\) 之前就死掉了（ 在 \(t\) 之類的 ）就代表說，
\(k+1\) 到 \(t\) 會把 第 \(k\) 條邊的兩端點連起來，也就是說，跟 \(k\) 的效果是一樣的。
這樣的話，第 \(i\) 條邊往前找的時候，就不會找到 \(k\) 而是 \(k+1\) 或更右邊的。

所以我們知道，\(k\) 跟 \(i\) 會是一對的。

感覺只有 3 分的做法

於是我們可以針對每一條邊往前找到 \(k\) 然後在他們的中點打一個 tag，表示要拔 \(k\) 換 \(i\)， 複雜度感覺是 \(O(M^2 + NQ)\)

但加了 io 優化就滿分了？？？

快，還要更快

在看了別人的 submission 後，我覺得我應該是假解，所以要來想怎麼優化。
